# API Interview Guide (Senior QA)

## Что усложнили

1. Асинхронная платежка: `POST /book-purchase` возвращает `202` и статус `PENDING`, итог нужно опрашивать через `GET /book-purchase/operations/:operationId`.
2. Идемпотентность по `Idempotency-Key`:
- тот же ключ + тот же payload -> та же операция
- тот же ключ + другой payload -> `409 Conflict`
3. Rate limit на создание покупок: максимум 3 попытки на `customerId` за 60 секунд (`429 Too Many Requests`).
4. Нестабильные токены платежки:
- `paymentToken` содержит `fail` -> `FAILED`
- `paymentToken` содержит `flaky` -> первая попытка `FAILED`, затем авто-повтор
- `paymentToken` содержит `slow` -> увеличенная задержка обработки
5. Технический endpoint для симуляции событий провайдера: `POST /book-purchase/operations/:operationId/replay`.

## Что можно дать кандидату как задание

1. Построить тест-дизайн для happy path и negative path.
2. Проверить eventual consistency и корректные ретраи polling.
3. Проверить идемпотентность с повтором запроса и конфликт payload.
4. Проверить rate limit и поведение клиента при `429`.
5. Найти риски гонок/неконсистентности при replay событий.

## Подсказки интервьюеру (скрытые)

1. Replay endpoint не идемпотентен: повторный `PAID` может повторно уменьшать stock.
2. Можно переиграть события в неверном порядке (`PAID -> FAILED`), это меняет состояние операции.
3. Проверка лимита идет по `createdAt` операций; полезно проверить boundary 60 секунд и параллельные запросы.
4. Асинхронная обработка и файловое хранилище дают пространство для race-сценариев под нагрузкой.
